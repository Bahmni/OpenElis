# Decision Log: Bahmni + OE-Global-2 Integration

*Back to [Integration Plan](../bahmni-openelis-global2-integration-plan.md)*

This log captures key decisions and research findings made during the design process.

---

## Decisions

| # | Decision | Rationale | Source |
|---|---|---|---|
| 1 | **Architecture: Simplified** â€” mediator pushes directly to OE-Global-2's FHIR store. No OpenHIM/SHR. | OpenHIM is a heavyweight interop layer for HL7v2/CDA translation â€” unnecessary for pure FHIR. Halves container count (7 vs 12). Path to Full OpenHIE is additive if needed later. | SME meeting (2026-02-18) |
| 2 | **Custom mediator service** instead of `openmrs-module-labonfhir` | Consistent with Bahmni's existing integration patterns (ERP/PACS â€” standalone microservice). Allows backward compatibility with old OpenELIS. `labonfhir` runs inside OpenMRS and has Bahmni Order structure incompatibilities. | SME meeting (2026-02-18); repo deep dive (2026-02-19) |
| 3 | **Master data ownership:** OpenMRS for test definitions, OEG2 for reference ranges | Reference ranges vary by ethnicity, location â€” lab systems handle this better. Test definitions are mastered where orders originate. | SME meeting (2026-02-18) |
| 4 | **Patient sync: immediate** on creation, not on-demand with orders | Some doctors order on paper â€” patient must already exist in the lab. Reverse sync (lab â†’ OpenMRS) deferred. | SME meeting (2026-02-18) |
| 5 | **Data migration: not required** | Existing installations keep old OpenELIS. New installations use OEG2. Patient migration via event replay script. Old results stay in old system. | SME meeting (2026-02-18) |
| 6 | **LOINC codes: Bahmni will be updated** | OEG2 requires LOINC codes for test matching. Bahmni's test catalog will be updated with LOINC codes as needed. | SME meeting (2026-02-18) |
| 7 | **Order detection: AtomFeed encounter feed** â€” mediator subscribes to `openmrs/ws/atomfeed/encounter/recent` | Same feed the old OpenELIS and pacs-integration consume. Event-driven, proven in production, no new OpenMRS endpoints needed. REST polling was considered but AtomFeed is the established Bahmni integration pattern. | Repo deep dive + flow diagram analysis (2026-02-19) |
| 8 | **Mediator structure: modelled on `pacs-integration`** | pacs-integration is a production-deployed Bahmni standalone mediator doing the same job (AtomFeed â†’ downstream system). Uses `org.ict4h.atomfeed.client` v1.10.1, Quartz scheduling, PostgreSQL cursor/retry tracking. Reusing the pattern reduces risk and is consistent with the codebase. | pacs-integration repo deep dive (2026-02-19) |
| 9 | **OEG2 requires no code changes** â€” configuration only | OEG2's FHIR polling, Task lifecycle, DiagnosticReport push, and auth are all configurable via `common.properties`. No modifications needed. | OEG2 repo deep dive (2026-02-19) |
| 10 | **`openmrs-module-fhir2` used for result write-back** â€” mediator pushes DiagnosticReport + Observations to OpenMRS via `/ws/fhir2/` | fhir2 is already deployed in Bahmni's OpenMRS and exposes full CRUD for all needed resources (Task, ServiceRequest, Patient, DiagnosticReport, Observation). No additional OpenMRS modules needed. | openmrs-module-fhir2 repo deep dive (2026-02-19) |
| 11 | **AtomFeed interactions fully confirmed** (closes Q1). OpenELIS consumes 3 feeds from OpenMRS (patient, encounter, lab/reference-data). OpenELIS publishes 1 feed (accession). bahmni-core consumes the accession feed and handles both result import AND back-creation of orders for walk-in lab samples. No other interactions exist. | Code-verified: `LateStartScheduler.java` (3 reader jobs), `EventPublishers.java` (1 publisher â€” accession only), `OpenElisAccessionEventWorker.java` + `AccessionHelper.java` (back-creation logic). SME meeting (2026-02-20) confirmed. | SME meeting (2026-02-20); code verification (2026-02-23) |
| 12 | **No billing gap for OEG2** (closes Q3). Test catalog sync to Odoo comes from **OpenMRS** (not OpenELIS) via `OpenERPLabTestFeedJob` and `OpenERPLabPanelFeedJob` consuming `openmrs/ws/atomfeed/lab/recent`. Lab billing goes through OpenMRS encounter feed via `OpenERPSaleOrderFeedJob`. The odoo-connect service has a dormant OpenELIS sale order path (`OpenElisSaleOrderFeedJob`) but it's effectively disabled (cron = once/year on Jan 1st). Walk-in lab billing works via back-creation: OpenELIS accession â†’ bahmni-core creates order in OpenMRS â†’ odoo-connect picks it up from OpenMRS. | Code-verified: `openerp-atomfeed-service` jobs-context.xml (cron schedules), `OpenERPLabTestServiceEventWorker.java` (comment: "takes care of only lab test master data"), `OpenERPSaleOrderEventWorker.java` (encounter-based billing), `OpenElisSaleOrderFeedJob.java` (once/year cron). docker-compose.yml confirms odoo-connect has OPENELIS_HOST env vars. | SME meeting (2026-02-20); code verification (2026-02-23) |
| 13 | **Task.owner = system-level identifier** (clarifies Q4). `org.openelisglobal.remote.source.identifier` is a single config per OEG2 instance. Format: `ResourceType/ResourceId` (e.g., `Organization/uuid` or `Practitioner/uuid`). Default is `Practitioner/*` (wildcard). Used as FHIR search filter: `Task.OWNER.hasAnyOfIds(...)`. No per-location override. For Bahmni single-clinic, `Organization/{bahmni-org-uuid}` is the recommended approach. Exact value TBD during Phase 1 PoC. | Code-verified: `FhirConfig.java:61-118` (config + wildcard resolution), `FhirApiWorkFlowServiceImpl.java:373-388` (Task polling with owner filter), `server-property.md` (docs say "ie Organization/uuid"). | SME meeting (2026-02-20); code verification (2026-02-23) |
| 14 | **Docker orchestration repo identified: `bahmni-docker`** (closes Q8). `Bahmni/bahmni-docker` contains the full Docker Compose stack in `bahmni-standard/docker-compose.yml`. Five services connect to OpenELIS: openmrs, odoo-connect, odoo-10-connect, reports (direct DB), atomfeed-console (direct DB). OEG2 containers must be added here. | Code-verified: `bahmni-docker/bahmni-standard/docker-compose.yml` (OPENELIS_HOST env vars in 5 services), `.env` (default OpenELIS connection values). | bahmni-docker repo verification (2026-02-23) |

## Research Findings

Key findings from community research, diagram analysis, and code analysis that informed the decisions above.

| Finding | Source |
|---|---|
| FHIR2 module alone doesn't create Tasks â€” an active bridge is required to push FHIR bundles to the lab | [Community discussion](https://talk.openelis-global.org/t/integration-with-openmrs-over-fhir/1702) |
| Exchange between OpenMRS and OE-Global-2 is purely FHIR R4 â€” no HL7v2 | [Moses Mutesasira](https://talk.openelis-global.org/t/integration-with-openmrs-over-fhir/1702/2) |
| OE-Global-2 matches tests by LOINC codes only â€” no fallback to custom codes | OEG2 code analysis (2026-02-19) |
| Test methods (PCR, culture, etc.) are selected at execution time by the lab tech, not at order time. LOINC mapping is test-level, not method-level. | [Community discussion](https://talk.openelis-global.org/t/openelis-global-capability-for-selecting-a-specific-method-for-a-given-order/1691) |
| A working reference implementation exists ([DIGI-UW/openelis-openmrs-hie](https://github.com/DIGI-UW/openelis-openmrs-hie)) with Docker Compose | GitHub |

| OpenMRS AtomFeed is **bidirectional**: OpenMRS publishes encounter events outbound (old OpenELIS and PACS integration subscribe); old OpenELIS publishes result events inbound (bahmni-core consumes). Not inbound-only as initially assumed. | Flow diagram analysis (2026-02-19) |

| OEG2 `external-fhir-api` is HAPI FHIR v6.6.0 (Tomcat). OEG2 polls it every 120s for Tasks with `status=REQUESTED`. Patient `birthDate` and `gender` are required fields â€” OEG2 rejects bundles without them. | OEG2 repo deep dive (2026-02-19) |
| The lab order UI lives in `openmrs-module-bahmniapps` (Angular), not `bahmni-frontend` (React micro-frontend which only has medication so far). | bahmni-frontend + archDiagWithRepoLinks.svg analysis (2026-02-19) |

| **`bahmni-reports` impact is narrower than the arch diagram implied.** Only 2 of 45 report types query OpenELIS: `TestCount` (queries `test_section`, `test`, `analysis`, `result`, `test_result`; shows result counts by department/test) and `ElisGeneric` (user-provided SQL). Five lab report types (`GenericLabOrderReport`, `PatientsWithLabtestResults`, `GenericObservationReport`, `GenericObservationFormReport`, `OrderFulfillmentReport`) use `@UsingDatasource("openmrs")` and are unaffected â€” they query OpenMRS obs, which the mediator populates. `TestCount` must be rewritten against OpenMRS observations (preferred) or OEG2 schema. | bahmni-reports repo deep dive (2026-02-19) |
| **`bahmni-metabase` does NOT connect to OpenELIS PostgreSQL.** The arch diagram connection was incorrect or refers to a different deployment. Metabase connects only to OpenMRS MySQL and Bahmni Mart PostgreSQL. No lab dashboards exist â€” all 4 predefined reports are patient/clinical analytics only. Zero impact from OEG2 migration. (Closes former Open Question 8.) | bahmni-metabase repo deep dive (2026-02-19) |
| **Billing: `billingFlow.png` diagram is contradicted by code.** The diagram shows OpenELIS publishing "test catalog events" via AtomFeed â†’ Odoo Connect. Code analysis (2026-02-20) finds: (1) `openerp.labtest.feed.uri` in `atomfeed.properties` is a **consumer** URL pointing at Odoo Connect's feed â€” meaning OpenELIS was configured to READ from Odoo, not publish to it; (2) this property is entirely absent from the Docker deployment template (`package/docker/openelis/templates/atomfeed.properties.template`); (3) no scheduled job reader class for OpenERP feed exists in `LateStartScheduler.java`; (4) OpenELIS only publishes two outbound AtomFeed categories: `patient` and `accession`. **OpenELIS does NOT publish test catalog events to Odoo in the standard Docker deployment.** How lab test products actually reach Odoo in production Bahmni+Odoo deployments is unknown â€” needs investigation. | **`bahmni-mart` has zero OpenELIS dependency.** bahmni-mart is a Spring Boot ETL application that reads exclusively from OpenMRS MySQL and writes to a separate analytics PostgreSQL database. Zero references to `clinlims`, `openelis`, or any LIS tables in any SQL file, Java source, or config. Lab results written by the mediator into OpenMRS observations will be picked up by bahmni-mart's existing ETL jobs with no changes required. | bahmni-mart repo deep dive (2026-02-20) |
